# AIBLI 搞怪截图 OCR 智能处理 - Automation Prompt

## 任务目标
扫描 `E:\搞怪\待处理\` 中的图片，识别文字内容，智能匹配配音角色，生成标准格式口播稿，并触发后端流水线自动执行。

## 输入
- 文件夹：`E:\搞怪\待处理\`
- 支持的图片格式：.jpg, .jpeg, .png, .gif, .bmp, .webp
- 排除子目录：`E:\搞怪\待处理\已处理\`

## 角色人设数据源
读取以下路径获取所有可用角色的人设信息：
- `D:\AIBLI\audio_synthesis\character_profile\{角色名}\personality.txt`（性格、声线、出处）
- `D:\AIBLI\audio_synthesis\character_profile\{角色名}\suitable_roles.txt`（适合配什么类型台词）

可用角色列表（15个）：玉玉、柊優花、W、希露菲成年版、希露菲青年版、卡夫卡、洛琪希、洛琪希2、白菜、知更鸟、管理员.B、索尔、艾尔黛拉、认知进化、陈千语

## 处理流程

### 第一步：扫描图片
1. 创建当天日期文件夹 `E:\搞怪\{YYYY-MM-DD}\`（如已存在则跳过）
2. 扫描 `E:\搞怪\待处理\` 中所有图片文件（排除 `已处理` 子目录）
3. 如果没有图片，直接结束并报告"今日无待处理图片"

### 第二步：识别图片文字
对每张图片使用 OCR 识别文字内容。如果某张图片识别失败，记录错误并跳过该图片，继续处理下一张。

### 第三步：内容分类判断
根据识别到的文字，判断每张图片属于以下哪种类型：

**类型A：多角色对话**
- 特征：2个以上不同说话人在对话，内容有意义
- 处理：按说话人切换拆分成多个txt文件，角色一换就新建文件
- 输出格式：每个文件内部格式为 `{角色代号}（第N段）\n----\n内容`
- 角色代号用数字：第一个出现的说话人=1，第二个=2，依此类推
- 文件名：`{关键词}_{序号}.txt`

**类型B：纯长文案/文章/小说**
- 特征：长篇文字、故事、文章，无明确多人对话
- 处理：单个txt文件，直接输出文案内容
- 文件名：`{关键词}_{YYYY-MM-DD}.txt`

**类型C：一句话梗/段子**
- 特征：简短搞笑句子、梗图配文
- 处理：单个txt，直接输出内容
- 文件名：`{关键词}_{YYYY-MM-DD}.txt`

**类型D：无价值内容**
- 特征：全是短水评/重复刷屏、只有表情包、官方通知、内容太短且无笑点
- 处理：跳过不保存

### 第四步：角色智能匹配（仅类型A需要）
1. 读取所有15个角色的 personality.txt 和 suitable_roles.txt
2. 分析每个说话人的台词风格（吐槽、元气、御姐、治愈、热血等）
3. 将说话人与最适合的角色进行匹配
4. 输出标准格式：`[角色名] 台词内容`

**匹配原则**：
- 吐槽役、搞笑担当 → 玉玉
- 元气少女、VTuber风格 → 柊優花
- 御姐、反派、女王 → 卡夫卡
- 治愈系、人妻、ASMR → 希露菲成年版
- 青梅竹马、纯真少女 → 希露菲青年版
- 师父、傲娇、知性 → 洛琪希/洛琪希2
- 萝莉、萌系、清楚系 → 白菜
- 歌姬、治愈、圣女 → 知更鸟
- 治疗师、学者、温柔 → 艾尔黛拉
- 硬汉、战斗狂、大叔 → 索尔
- 旁白、叙事主角 → W
- 系统提示、AI助手 → 管理员.B
- AI解说、科技评论 → 认知进化
- 女侠、战斗系、古风 → 陈千语

### 第五步：生成标准格式文件
**类型A输出格式示例**：
```
[玉玉] 这是第一句台词
[柊優花] 这是第二句台词
[玉玉] 这是第三句台词
```

**类型B/C输出格式**：直接输出文案内容，不需要角色标记

**双通道输出**：
- 原始文本保存到：`E:\搞怪\{YYYY-MM-DD}\{文件名}.txt`
- 标准格式同时保存到：`D:\AIBLI\pipeline\queue\pending\{文件名}.txt`

**去重规则（防止重复处理）**：
1. 写入 `D:\AIBLI\pipeline\queue\pending\` 前，先检查该目录是否已有**同名文件**。如果有，说明 pipeline 尚未处理完，**跳过不再重复写入**。
2. 同时检查 `D:\AIBLI\pipeline\queue\done\` 是否已有同名文件。如果有，说明该内容已处理完毕，**跳过不写入**。
3. 只有 pending 和 done 都不存在同名文件时，才允许写入。
4. 报告摘要中标注"跳过了 X 张（重复/已处理）"

### 第六步：图片归档
将处理完的图片移动到 `E:\搞怪\待处理\已处理\`

### 第七步：输出处理摘要
报告本次处理结果：
- 处理了几张图片
- 生成了几个文件（按类型统计）
- 跳过了几张（及原因）
- 角色匹配结果

### 第八步：触发后端流水线
1. 检查是否有新生成的文件进入 `D:\AIBLI\pipeline\queue\pending\`
2. 如果有，启动 pipeline GUI：`python D:\AIBLI\pipeline\pipeline_gui.py`
3. 如果 GUI 已经在运行（通过进程检查），则不重复启动
4. GUI 会自动扫描 pending 队列并逐个串行执行（文案→音频→视频）

## 通用规则
1. 不要输出：日期时间、点赞数、浏览量、作者等级、UI元素文字
2. 不要输出：【第X格】【画面描述】等技术性标注
3. 不要输出："这是一张截图"等说明性文字
4. 口语化改写：书面语太重的改成自然口语，加少量语气词（啊、嘛、呢、吧），保持原意不变
5. 文件命名用内容关键词，去掉特殊字符和空格
6. 如果识别到的文字为空或无法读取，记录错误并跳过

## 错误处理
- 单张图片处理失败 → 记录错误，继续处理下一张
- 角色人设文件读取失败 → 使用默认角色（玉玉、柊優花循环分配）
- Pipeline GUI 启动失败 → 记录错误，报告用户手动启动
- 任何步骤出错都要记录到日志，不要中断整个流程
